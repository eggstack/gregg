# Phase 11: version 1.0.1 corrective closure

## Objective

Correct the release-blocking defects discovered after the `1.0.0` implementation attempt, replace unsubstantiated completion claims with reproducible evidence, and publish a narrowly scoped `1.0.1` patch release of all affected crates.

This is a correctness and release-evidence pass. It must not broaden Gregg into a general monitoring platform, add public-internet hardening, introduce historical telemetry, or redesign the three-crate architecture.

The existing architecture remains the intended architecture:

```text
gregg-protocol  ◄── greggd
gregg-protocol  ◄── gregg
```

The implementation is already substantial. The purpose of this phase is to make the shipped behavior match the documented version-1 contract on Linux and macOS, close lifecycle and configuration defects, and prove the result on the supported target classes.

## Release disposition

`1.0.0` has already been published and cannot be replaced. All source corrections must use a new patch version.

Before beginning publication work:

1. Record whether `gregg`, `greggd`, and `gregg-protocol` `1.0.0` remain unyanked or are yanked.
2. Base that decision on confirmed impact, not on convenience.
3. At minimum, confirm the native macOS daemon behavior using the published `greggd 1.0.0` source or package.
4. If `greggd 1.0.0` cannot produce valid native macOS snapshots, yank `greggd 1.0.0` and document the reason. Consider yanking `gregg 1.0.0` only if a client defect is severe enough to make ordinary configuration unsafe or materially misleading.
5. Do not yank `gregg-protocol 1.0.0` unless a protocol-level defect is independently demonstrated.
6. Do not move or recreate the `v1.0.0` tag.

The corrective release target is `1.0.1`. If publication state or an already-uploaded patch version prevents using `1.0.1`, stop and document the state before selecting another version.

## Scope and priority

### P0: release blockers

The following must be corrected before any new release:

- macOS Mach host-port acquisition and lifetime.
- macOS `xsw_usage` ABI layout.
- Native macOS collector smoke coverage.
- Daemon task supervision and startup failure propagation.
- Wiring `stale_after_ms` into the HTTP serving policy.
- launchd start/stop/restart/croncheck correctness.
- Linux and macOS installer default-path correctness.
- Client default-port and explicit-port semantics.
- Effective cross-process configuration locking.
- True bounded HTTP response-body reads.
- Reproducible package, install, loopback, and native-platform evidence.

### P1: version-1 contract completion

These should land in the same pass unless a documented platform constraint blocks them:

- Immediate first polling generation.
- Functional `RefreshNow` wiring.
- Systemd non-root service execution or an explicit, justified exception.
- CI coverage for all package archives, MSRV, dependency policy, and native smoke tests.
- Documentation corrections and plans-registry reconciliation.

### Non-goals

Do not add:

- TLS or authentication.
- Public-internet deployment support.
- Per-process metrics.
- Historical storage.
- Alerting.
- Remote command execution.
- Service discovery.
- Plugin systems.
- New metrics beyond the established version-1 schema.
- A rewrite of the HTTP, polling, state, or rendering architecture.

## Workstream A: repair the native macOS collector

### A1. Obtain a valid Mach host port

The production FFI currently passes a literal zero as the `host_t` argument to host-statistics APIs. Replace that with a valid host port obtained through the supported Mach host-self interface.

Required implementation properties:

- Declare the correct host-self function or use a narrowly scoped, well-maintained binding that does not expand dependencies unnecessarily.
- Obtain the host port before calling `host_statistics`, `host_statistics64`, or `host_page_size`.
- Use the same valid host port for the related query sequence where practical.
- Follow the Mach port ownership rules. If the returned right must be deallocated, contain that behavior inside the FFI module and make cleanup unconditional.
- Do not expose raw Mach ports outside `collector/macos/ffi.rs`.
- Validate every Mach return code and returned count.
- Preserve the existing safe owned-record interface consumed by the collector.

Prefer a small RAII wrapper such as:

```rust
struct HostPort(mach_port_t);

impl HostPort {
    fn current() -> Result<Self, CollectError>;
    fn raw(&self) -> mach_port_t;
}
```

If deallocation is required, implement it in `Drop`. Do not let panic paths leak ownership assumptions.

### A2. Correct `xsw_usage`

Define the Darwin swap structure with the exact field order and compatible C widths:

```text
xsu_total
xsu_avail
xsu_used
xsu_pagesize
xsu_encrypted
```

Requirements:

- Include `xsu_avail`; do not omit or reorder fields.
- Validate the returned byte length against the exact expected structure size.
- Reject short or otherwise incompatible results as typed parse/source errors.
- Map `xsu_total` and `xsu_used` into `RawSwapUsage`.
- Keep clamping and percentage normalization outside the FFI layer.
- Add compile-time layout assertions where stable Rust permits them without fragile platform assumptions.
- Add a unit test for the expected field mapping using a synthetic fully populated structure.

### A3. Strengthen native identity and memory failure handling

The existing implementation silently substitutes `1` logical core and `0` physical memory when integer sysctls fail. For native production queries:

- A failure to read logical CPU count must be a typed collection error unless a separately documented, reliable fallback exists.
- A failure to read physical memory must be a typed collection error; publishing a zero memory total is invalid.
- Validate `sysctlbyname` returned lengths before interpreting generic integer storage.
- Ensure string sysctl reads handle length changes between the sizing and read calls without out-of-bounds indexing.
- Keep product-version fallback behavior explicit and documented. Kernel release, architecture, logical cores, physical memory, CPU, memory, swap, and load are required for a ready version-1 sample.

### A4. Add native macOS smoke tests

Mock-based tests remain useful but are not sufficient for FFI correctness.

Add a native test gated with `#[cfg(target_os = "macos")]` that:

1. Constructs `MacOsCollector::new`.
2. Takes the warming sample.
3. Waits only the minimum deterministic interval needed for CPU counters to advance.
4. Takes a second sample.
5. Builds and validates a complete protocol snapshot.
6. Asserts:
   - logical cores are at least one;
   - physical memory is nonzero;
   - memory used does not exceed memory total;
   - swap used does not exceed swap total;
   - load values are finite and nonnegative;
   - CPU percentage is finite and within `0..=100`;
   - I/O wait is unsupported and null;
   - identity reports Darwin/macOS and a nonempty architecture.

Also add direct native query tests for:

- CPU load info.
- VM info and page size.
- Swap usage.
- Load averages.
- Required identity sysctls.

These tests must execute on both the Intel macOS and Apple Silicon CI jobs. They may not be replaced by mocks on those jobs.

### A acceptance criteria

Workstream A is complete when:

1. No host-statistics call receives a fabricated Mach port.
2. The swap ABI matches Darwin exactly and validates returned length.
3. Required native identity values do not silently become zero or synthetic defaults after query failure.
4. Production `FfiNativeQueries` is exercised by native tests on Intel and Apple Silicon macOS CI.
5. A real `greggd run` reaches ready state on both macOS architectures and `/v1/status` validates.
6. Native values are compared against diagnostic references (`sysctl`, `vm_stat`, `top`, or Activity Monitor) with documented tolerances; diagnostic commands remain test/manual references, not runtime dependencies.

## Workstream B: daemon runtime supervision and freshness

### B1. Fail startup when binding fails

Do not detach a server task and then wait only for an operating-system signal.

Refactor startup so one of these equivalent guarantees holds:

- The TCP listener is bound before long-running sampler/server tasks are spawned; or
- The main supervisor selects over the server task, sampler task, and shutdown signal and treats unexpected task termination as a daemon failure.

Required behavior:

- Address-in-use produces a nonzero exit promptly.
- Invalid/unavailable bind addresses produce a nonzero exit promptly.
- Sampler panic or unexpected termination produces a nonzero exit.
- HTTP server runtime failure produces a nonzero exit.
- Normal SIGTERM/SIGINT causes orderly cancellation and a zero exit.
- Task errors and join errors are not discarded.
- The daemon never remains alive while no HTTP listener exists.

Prefer explicit task ownership and cancellation. Do not use detached tasks whose failures cannot reach the process supervisor.

### B2. Wire `stale_after_ms`

`Config::stale_after_ms` must control the server’s snapshot-age policy.

Required behavior:

- Convert the validated millisecond value to `Duration` without truncation.
- Construct `ServerState` with the configured stale policy rather than always using `ServerState::new()`.
- Decide whether failure-count staleness remains disabled in version 1 or receives a config field; do not add an undocumented fixed threshold.
- A valid cached snapshot may continue to be served after a transient collector error only until it exceeds the configured age.
- Once stale, `/v1/status` returns `503` with a typed health response.
- `/healthz` returns `503` while failed or stale.
- A subsequent valid sample restores `200` behavior and resets failure state.
- `stale_after_ms = 0` explicitly disables age-based staleness.

### B3. Preserve useful failure diagnostics

The sampler currently collapses server failure text to a generic message. Keep diagnostics concise and safe, but preserve the collection category and a bounded message in health state or logs.

Do not expose internal paths, backtraces, or unbounded upstream text over HTTP.

### B4. Add daemon supervision and stale-data tests

Add tests for:

- Port already occupied at startup.
- Server task failure after startup.
- Sampler task panic/failure propagation through a synthetic collector.
- Normal shutdown waits for both tasks.
- Snapshot served during an allowed transient failure window.
- Snapshot rejected immediately after crossing `stale_after_ms`.
- Recovery after stale failure.
- Staleness disabled with zero.
- Wall-clock movement handled through saturating arithmetic.

At least one test must exercise the actual `run` supervision boundary rather than only `ServerState` methods.

### B acceptance criteria

1. Binding failure cannot leave a living sampler-only daemon.
2. Unexpected server or sampler termination reaches the process exit path.
3. Configured staleness is applied exactly and verified end to end.
4. Cached stale data cannot be returned indefinitely as healthy `200` data under the default configuration.
5. Normal service-manager shutdown remains graceful.

## Workstream C: service management and installers

### C1. Define launchd lifecycle semantics

Use launchctl domain targets and service targets correctly:

- Domain target for loading a plist: `system`.
- Service target for an already loaded job: `system/com.eggstack.greggd`.
- Plist path: `/Library/LaunchDaemons/com.eggstack.greggd.plist`.

Implement lifecycle commands with coherent semantics:

- `start`: if loaded but stopped, kickstart it; if not loaded, bootstrap from the installed plist.
- `stop`: stop the service without making a subsequent `start` impossible. Choose either a kill/disable model or deliberately bootout and ensure `start` re-bootstraps.
- `restart`: restart a loaded service; if installed but unloaded, bootstrap it.
- `croncheck`: if active, no-op; if installed but inactive/unloaded, make exactly one bounded start/bootstrap attempt; if the plist is missing, return a clear nonzero error rather than looping.
- `is_active`: distinguish loaded, running, and absent states where launchctl allows it. Do not parse only an ambiguous substring.

Centralize command construction and make it injectable so tests verify the executed program, argument vector, and state transitions without calling the host service manager.

### C2. Correct installer paths and behavior

Both install scripts must derive the repository root as the parent of `packaging/`, not the parent of the repository.

Requirements:

- Default binary path resolves to `<repo>/target/release/greggd`.
- An explicit binary path continues to work.
- Help/header text matches actual behavior.
- Decide whether installers start the service. If they do not, state that consistently. If they do, make the action explicit and idempotent.
- Preserve existing valid config files.
- Validate newly created configs by invoking `greggd run` validation or an explicit config-check path if one is added.
- Quote all paths containing spaces.
- Validate launchd plist syntax with `plutil -lint` on macOS.
- Validate systemd unit syntax with `systemd-analyze verify` where available.

### C3. Run the Linux daemon unprivileged

The daemon does not require root for metrics or its default port. Choose and implement one supported model:

- a dedicated system user created by the installer; or
- `DynamicUser=yes` with compatible state/config access.

The selected model must allow:

- reading the daemon config;
- reading `/proc` and required kernel interfaces;
- binding the configured unprivileged address/port;
- writing only the intended log/state paths, if any;
- `greggd host` and `greggd port` to mutate configuration through an appropriately privileged CLI invocation and then restart the service.

Document the ownership and permission model. Do not leave the daemon running as root merely because the installer itself requires root.

### C4. Service and installer tests

Provide deterministic command-runner tests for launchd and systemd adapters. Add shell tests, Bats tests, or an equivalent portable harness for installer path resolution and generated configuration.

Native/manual evidence must include:

- Linux systemd install, start, stop, restart, croncheck, host mutation, and port mutation.
- macOS launchd install, start, stop, restart, croncheck, host mutation, and port mutation.
- Config and service behavior after a reboot-equivalent reload/bootstrap.

### C acceptance criteria

1. `stop` followed by `start` works on launchd.
2. `croncheck` can recover an installed but inactive daemon on both service managers without looping.
3. Installer default paths resolve correctly from a clean checkout.
4. Installer documentation matches actual start/bootstrap behavior.
5. The Linux service runs under a non-root identity unless a documented, reviewed platform blocker proves this impossible.
6. Native lifecycle evidence is recorded for systemd and launchd.

## Workstream D: client configuration correctness

### D1. Preserve whether a port was explicit

The endpoint parser must distinguish:

```text
host             -> no explicit port
host:11310       -> explicit port 11310
host:11320       -> explicit port 11320
[::1]            -> no explicit port
[::1]:11310      -> explicit port 11310
```

Do not infer explicitness by comparing the resulting value with the built-in default.

A suitable model is:

```rust
struct EndpointSpec {
    host: String,
    port: Option<u16>,
}
```

or a separate `port_was_explicit` field.

Required behavior:

- `gregg add host` stores the current configured `default_port`.
- `gregg add host:11310` stores exactly `11310`, even when `default_port` differs.
- `gregg remove host` removes every endpoint for that normalized host.
- `gregg remove host:11310` removes only that exact host/port.
- IPv4, DNS, mDNS, and IPv6 forms retain equivalent semantics.
- Duplicate detection uses normalized host plus the final effective port.

### D2. Implement a real cross-process lock

`ConfigStore` mutation must serialize independent CLI processes, not only threads in one process.

Requirements:

- Retain the file descriptor or file handle for the full critical section.
- Hold the operating-system lock across load, mutation, validation, atomic rename, and verification.
- Release the lock through RAII.
- Use a stable adjacent lock path.
- Bound lock waiting or return a clear busy error; do not wait forever silently.
- Do not delete another process’s active lock file.
- Keep the in-process mutex only if useful in addition to, not instead of, the OS lock.
- Ensure advisory-lock tests prove that a second process or independently opened handle cannot acquire the lock concurrently.

Use a small focused dependency if it materially reduces unsafe platform code. Any new dependency must pass license/advisory review and remain appropriate for Linux and macOS.

### D3. Correct atomic-write durability

The existing write-flush-rename pattern should additionally:

- use collision-resistant temporary names rather than only PID-derived names;
- preserve or set intentional file permissions;
- sync the containing directory after rename where supported and useful;
- re-open and validate the final file;
- clean abandoned temporary files safely without deleting unrelated files.

### D acceptance criteria

1. Configured default ports are honored for portless additions.
2. An explicitly supplied default-valued port remains explicit for removal.
3. Host-only and host/port removal semantics are unambiguous and fixture-tested.
4. Two concurrent CLI processes cannot lose one another’s config mutations.
5. Interrupted or failed writes preserve the previous valid configuration.

## Workstream E: bounded polling and event-loop completion

### E1. Enforce the body cap while streaming

Do not call an unbounded `response.bytes()` and check size afterward.

Required behavior:

- Reject immediately when a valid `Content-Length` exceeds the cap.
- Stream chunks while tracking cumulative length.
- Stop reading and return `BodyTooLarge` as soon as the cap is exceeded.
- Preallocate only within the cap.
- Handle absent, incorrect, or malicious `Content-Length` values safely.
- Preserve timeout and cancellation behavior while streaming.

Add tests with:

- oversized declared content length;
- chunked/unknown-length body that crosses the cap;
- exact-cap body;
- cap-minus-one body;
- partial body followed by close;
- cancellation during streaming.

### E2. Poll immediately on startup

The first generation should begin immediately after the TUI starts. Sleep after a completed generation or schedule against an explicit next deadline.

Required behavior:

- No default five-second blank/pending period before the first request.
- Refresh timing does not accumulate unbounded drift.
- Slow generations follow the documented skip-if-running behavior.
- Cancellation remains prompt during sleep and active requests.

### E3. Wire `RefreshNow`

Add an explicit control channel or scheduler command interface.

Required behavior:

- Ctrl-R starts an immediate generation.
- It respects the overlap policy and concurrency limit.
- It does not create concurrent unbounded generations.
- It does not reset configured periodic timing in a way that causes rapid duplicate cycles.
- The action is tested through the actual event-loop/scheduler boundary, not only through enum construction.

### E4. Preserve endpoint membership across task failures

If a polling task panics or is cancelled, the resulting batch must still account for the endpoint with a typed `Cancelled` or internal-failure outcome. Do not silently omit the endpoint from the batch.

Prefer structured concurrency (`JoinSet`, `FuturesUnordered`, or equivalent) that retains endpoint identity alongside the task.

### E acceptance criteria

1. The response body cap bounds actual bytes read and allocated.
2. The first polling generation begins immediately.
3. Ctrl-R causes a real bounded refresh.
4. Every configured endpoint produces a result in every completed generation unless the whole generation is cancelled.
5. Shutdown cancels or joins all active polling work before terminal restoration completes.

## Handoff rule

The implementing agent should make narrowly scoped commits organized by workstream. Each commit must update tests and documentation for the behavior it changes. Do not combine unrelated refactors with corrective work.

If native hardware or service-manager access blocks an acceptance criterion, record the exact blocker and leave the criterion open. Do not substitute mock coverage, script presence, or prospective commands for native evidence.
